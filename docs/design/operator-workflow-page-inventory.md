# Operator workflow page inventory

Purpose: define the workflow-first operator page set before any final page drafting. Each page below is an operator-language contract, not a UI implementation. The intended pages should explain what work the workflow removes, what source data the deterministic app must provide, what the agent may draft or rank, what a human must approve, how labor/outcome evidence is measured, and which repo/Rustdoc surfaces should back the claims.

Source lens:

- `nva-pet-resorts-ai-context.md`: NVA Pet Resorts is a 170-site, multi-brand operating portfolio where labor savings come from workflow automation rather than generic chat.
- `docs/design/labor-cost-reduction-crosswalk.md`: agents consume app-owned context packets and submit reviewable drafts/recommendations; deterministic app contracts own facts, policy, review gates, audit, side effects, and outcome capture.
- `README.md` and `app/README.md`: READMEs are navigation/wiki surfaces; executable examples belong in Rustdoc/doctests on `domain`, `app`, `storage`, and `gingr` public APIs.

## Standard page contract

Use the reusable [`entity-driven workflow page template and evidence matrix`](entity-driven-workflow-page-template.md) before drafting any final workflow page. It expands this inventory into the page skeleton, citation format, featured-entity/contract matrix, and evidence-gap rules.

Every operator workflow page must answer these twelve questions in this order:

1. What problem does this solve?
2. Whose time does it save?
3. What source data does it need?
4. Which entities are featured?
5. Which related entities should be mentioned without becoming the page center?
6. Which app/domain/storage/provider contracts are featured?
7. Who or what is authoritative for each fact or decision?
8. What does the agent draft, rank, recommend, validate, summarize, or record?
9. What must a human approve?
10. What actions are blocked or human-reviewed by default?
11. What outcome or labor value gets measured?
12. What code/Rustdoc/test evidence backs this up?

## Shared source-of-truth and page-shape rules

- Pages are operator workflow pages, not Rust API docs. They should explain the review queue in manager/front-desk/regional language and link to Rustdoc/code as evidence.
- Each page should name the context packet, draft/recommendation packet, approval gate, blocked actions, and outcome record before discussing UI copy.
- Source data must be app-owned packets or normalized source/read-model facts with `domain::source::RecordRef`/`Provenance`; raw provider payloads, model memory, prior AI summaries, or unredacted notes are not business truth.
- Agents may summarize, rank, draft, and recommend review queues. They must not execute customer sends, provider/PMS writes, schedule changes, payment/refund/discount movement, policy exceptions, source hiding, or sensitive payload exposure.
- Labor-savings claims must point to outcome capture. If a workflow lacks outcome storage today, the page should say so and name the intended outcome record instead of asserting savings.
