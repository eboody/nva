# Entity atlas inventory

Purpose: source-of-truth inventory for later entity-atlas and relationship-page writers. This is not a final atlas page set. It lists the visible entities that should get their own page, a relationship entry, or be merged into a larger page, and it explains the choice in plain pet-resort language.

Relationship artifact: [Entity atlas relationship map](entity-atlas-relationships.md) connects the drafted family pages into the operating-model edges from source evidence to packets, review gates, outcomes, storage proof, and runtime shells. Evidence artifact: [Entity atlas evidence-link anchors](entity-atlas-evidence-link-anchors.md) names source, Rustdoc, and test anchors plus explicit gaps for major entity/workflow entries.

Scope inspected:

- Rust source: `domain/src`, `app/src`, `storage/src`, `integrations/gingr/src`, `apps/api`, `apps/worker`, and `apps/cli`.
- Existing docs: root `README.md`, crate READMEs, service-line READMEs, `docs/design/*`, `docs/workflows/*`, `docs/architecture/*`, `docs/integrations/gingr/*`, `docs/ops/*`, and launch/audit/planning docs visible under `docs/`.
- Rustdoc source contracts: module/item doc comments in source files. No rendered `target/doc/*.html` Rustdoc tree was present in this workspace during this pass, so the Rustdoc locations below are source-backed module/type paths rather than rendered URLs.

Selection rule:

- Must-document: a visible concept that a non-coder operator, docs writer, or reviewer must understand to trace labor savings, source authority, safety review, provider mapping, or outcome evidence.
- Optional/merge: a supporting scalar, code table, adapter helper, or implementation shell that should be mentioned in a relationship map but probably does not need its own standalone page.

Likely workflow labels used below:

- Manager Daily Brief
- Booking Triage
- Data Quality Hygiene
- Checkout Completion
- Retention/Grooming Rebooking
- Daily Updates
- Regional Exceptions
- Gingr Source Normalization
- Runtime/API Operations

## Cross-family relationship notes for later writers

1. Provider facts flow from `integrations/gingr` endpoint/response/DTO/webhook surfaces into mapping candidates and `domain::source::Provenance`; they do not become domain truth by themselves.
2. Domain entities (`Location`, `Customer`, `Pet`, `Reservation`, documents, vaccines, incidents, messages, care notes, policy gates, workflow events) are the semantic truth that app workflows compose.
3. App workflow packets (`Booking Triage`, `Checkout Completion`, `CRM Retention`, `Daily Update`, `Manager Daily Brief`, `Data Quality Hygiene`) are operator review queues and draft/rank/summarize surfaces, not live side-effect executors.
4. Blocked actions and review gates should be documented beside every workflow packet because they are the safety story: no autonomous sends, PMS/Gingr writes, schedule moves, payments/refunds/discounts, policy exceptions, medical/safety approvals, or source hiding.
5. Storage records are proof/projection shapes. Outcome records and labor-minute values are the bridge from "this might save time" to measured manager/front-desk/reconciliation labor savings.
6. Runtime shells (`apps/api`, `apps/worker`, `apps/cli`) expose or run these contracts safely; they should not be described as owning alternate entity models.
