# Workflow-to-entity navigation map

Purpose: readers may arrive through a workflow because that is how resort labor is organized, but the repo's operating model remains entity-first. Use this map to start with a labor-saving workflow, jump into the relevant entity family pages, then come back to the workflow packet, review gate, outcome record, and source/Rustdoc evidence.

This page preserves the root README maxim: the repo is an entity-first operating model for lowering labor cost across NVA Pet Resorts. Workflow pages are wayfinding and examples. Entity pages, source contracts, review gates, outcome records, and evidence links are the spine.

## Entity-first reading loop

1. Start with the workflow row below that matches the operator job.
2. Open the linked entity family pages before reading the workflow as a procedure.
3. Check which entities are created, read, updated, or measured by the workflow.
4. Check the review gate and blocked-action family before trusting any draft or recommendation.
5. Check the outcome/storage evidence before treating a labor-saving claim as measured.
6. Use the source paths, tests, and Rustdoc/module paths as behavioral authority.

## Contract-reading guardrails

- Workflow prose must not replace entity pages. If a workflow needs a new fact, add or link the entity entry first.
- A workflow packet is not a system-of-record action. It is a reviewable bundle of source evidence, policy gates, safe actions, blocked actions, drafts, and outcomes.
- Source/provider facts are not domain truth until mapping, review, or policy contracts promote them.
- Storage records are proof/projections. They do not authorize live customer, provider, schedule, payment, or safety actions.
- Labor value is measured only when outcome records, actual minutes, staff disposition, and source refs exist.
- Planned/future workflows, especially regional labor exceptions, must stay labeled planned until a source/test/app contract exists.
