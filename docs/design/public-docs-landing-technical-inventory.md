The repository currently has three documentation surfaces:

1. The maintainer wiki in Markdown:
   - Root orientation: [`../../README.md`](../../README.md)
   - Design/architecture/ops/launch docs under [`../`](../)
   - Required README coverage is enforced by [`../../scripts/check_markdown_links.py`](../../scripts/check_markdown_links.py).

   - Current live title: `NVA Rustdocs`.
   - Current live body says the docs are generated from `eboody/nva` and refreshed automatically from `main`.
   - Current live root links directly to Rustdoc crate roots: `/domain/`, `/app/`, `/storage/`, `/gingr/`, `/pet_resort_api/`, `/pet_resort_worker/`, and `/cli/`.
   - That root page is not currently represented by a checked-in `index.html`, `mdBook`, `docs/` static-site config, GitHub Pages workflow, Netlify/Vercel config, `Makefile`, or `justfile` in this checkout.

## 2. Candidate implementation approach

Implement a checked-in, non-coder landing page source and a small reproducible build wrapper, while leaving Rustdoc output generated by Cargo.

Recommended shape:

1. Add a checked-in static landing source, for example:
   - `docs/public/index.html`
   - optionally `docs/public/styles.css` if the page needs more than inline minimal styles.
2. Add a repo-local build wrapper that makes the public docs artifact reproducible, for example:
   - `scripts/build_public_docs.sh`
3. The wrapper should:
   - run `cargo doc --workspace --no-deps` or call the existing Rustdoc gate when appropriate;
   - copy `docs/public/index.html` into `target/doc/index.html` after Rustdoc renders;
   - copy any `docs/public/*` static assets into a namespaced path such as `target/doc/public/` if needed;
   - avoid modifying generated Rustdoc crate pages.
4. Keep the landing page written for non-coders:
   - lead with labor-cost-reduction workflows and safety/review boundaries;
   - provide audience paths such as “manager/operator”, “technical reviewer”, and “integration/source-data reviewer”;
   - link to Rustdoc crate roots as evidence, not as the first thing a non-coder must decode.
5. Preserve direct generated Rustdoc links as stable evidence links:
   - `/domain/` for semantic operating contracts;
   - `/app/` for review-gated agent/workflow contracts;
   - `/storage/` for persistence/read-model boundaries;
   - `/gingr/` for provider adapter/source boundaries;
   - `/pet_resort_api/`, `/pet_resort_worker/`, and `/cli/` for runtime shells.

This approach makes the root page source-controlled and reviewable without hand-editing any `target/doc/**` generated output. It also gives the publish job one deterministic command to run before syncing the docs directory.

## 5. Risks and caveats

- The live root page currently exists, but its source is not checked into this repo. If an external deploy script owns that page, repo changes alone will not update <https://nva.eman.network/> until that deploy path is changed to use the new wrapper or checked-in static source.
- `cargo doc --workspace --no-deps` does not appear to create a root `target/doc/index.html` in this checkout. The root landing page should therefore be overlaid after Rustdoc generation, not expected from Rustdoc itself.
- Do not hand-edit `target/doc/**`; it is generated and will be overwritten by the next `cargo doc` run.
- Keep public landing copy distinct from source-derived Rustdoc contracts. The landing page should orient non-coders and link to evidence; it should not duplicate executable Rust examples or drift-prone API details.
- The Markdown link checker validates `*.md` local links, not HTML links. If the landing is HTML, add a focused smoke/test for required HTML hrefs if the implementation card wants automated coverage.
- Shared-checkout serialization matters. This task and its child implementation card use `/home/eran/code/nva` as a shared directory, not an isolated worktree. Before editing, implementation workers should check `git status --short` and avoid overwriting concurrent changes from docs/review cards.
- If the deploy service expects exactly the current live root link list, preserve backwards-compatible crate-root paths. Add friendlier audience navigation above them rather than removing the Rustdoc evidence links.
