# Entity atlas evidence-link anchors

Purpose: stable citation map for entity-atlas writers. Use this file when a family page needs source, Rustdoc, or test evidence without re-discovering the codebase. It is documentation-supporting evidence only; it does not create product behavior.

Scope checked from repo root: `domain/src`, `app/src`, `storage/src`, `integrations/gingr/src`, `apps/api`, `apps/worker`, `apps/cli`, and the current `target/doc` Rustdoc tree.

## How to cite evidence safely

- Prefer source paths for durable Markdown links. Rendered Rustdoc under `target/doc/` is useful local evidence but is regenerated and not a product claim by itself.
- If a rendered item page exists, cite it as local generated evidence; otherwise cite the Rustdoc/module path plus the source file.
- Treat tests as contract evidence. Link the specific test file that proves the safety boundary, source grounding, validation, or outcome persistence claim.
- If no test, rendered Rustdoc, storage projection, or live integration proof exists, say so explicitly instead of filling the gap with product copy.

## Verification commands

Run from repo root:

```sh
cargo doc --workspace --no-deps
python scripts/check_rustdoc_completeness.py
python scripts/check_markdown_links.py
cargo test -p app --test booking_triage_mvp --test checkout_completion_workflow_contracts --test crm_retention_workflow_contracts --test daily_care_update_mvp --test manager_daily_brief_workflow_contracts --test data_quality_hygiene_workflow_contracts
cargo test -p storage
cargo test -p gingr
cargo test -p pet-resort-api
cargo test -p pet-resort-worker
```

`python scripts/check_rustdoc_completeness.py` is the strongest single Rustdoc evidence command because it runs the strict missing-docs gate and smoke-checks representative generated pages.

## Evidence gaps that should remain explicit

- Rendered Rustdoc is local/generated under `target/doc`; final public docs should not claim a published Rustdoc URL until a published docs site exists.
- Live Gingr/PMS/POS writes, payment/refund/discount movement, booking mutations, schedule mutations, and customer sends are not proven by the atlas evidence and remain blocked or approval-gated.
- CRM retention has an app-level `OutcomeRecord`; durable storage projection for CRM conversion/revenue outcomes is not present in the current `storage::operations` surface.
- Production labor savings and regional BI are not proven by source contracts alone. Cite manager/data-quality outcome records or label claims as planned.
- Message state has source/Rustdoc evidence (`domain/src/message.rs`, `domain::entities::Message`, app daily-update/CRM/tool messaging contracts) and workflow tests, but any standalone operator atlas page should still make approval status, body refs, channel permission, and send gates explicit rather than treating “message” as arbitrary text.
- Provider surface gaps for undocumented grooming/training DTOs should cite `integrations/gingr/src/dto/*` gap markers and should not invent provider-backed fields.
