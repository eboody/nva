## Public-doc phrasing recommendations

- Use “business model / source-of-truth vocabulary” when explaining `domain`, but keep the code path nearby: `domain::*`.
- Use “workflow/review bundle” for `packet` on non-coder pages, with the Rust term in parentheses on first mention.
- Use “staff-review draft” or “draft-only artifact” instead of bare “draft” when customer messages, booking confirmations, payments, or provider records are nearby.
- Use “requires approved human/system action” for blocked actions to avoid implying product failure.
- Use “provider payload shape (DTO)” and “provider-native record” for Gingr data before mapping.
- Use “lineage/evidence trail” for provenance, and “source pointer” for `RecordRef`.
- Keep these pairings explicit in public docs: `domain` is not a database/API; `app` is not live automation; `storage` is not business truth; `integration` is not source-of-record; `DTO/provider record` is not domain truth; `read model` is not source-of-record; `draft` is not sent/applied; `outcome capture` records reviewed results rather than performing the result.
