# Kanban closeout checklist

Use this checklist before completing an NVA Kanban card or asking for owner review. The goal is to make stale state, noisy artifacts, inconsistent generated files, and overclaims visible while the worker still owns the workspace.

## Required handoff fields

Copy this shape into the board completion metadata or owner-review comment and fill every line with concrete evidence. If an item does not apply, say why.

```text
Changed-file classification:
- Source/runtime files:
- Tests:
- Documentation/runbooks:
- Generated or snapshot artifacts:
- Fixtures/synthetic data:
- Build/cache/noisy artifacts removed:

Generated artifact disposition:
- Keep:
- Regenerated by:
- Verified against source/schema by:
- Reverted or removed:

Stale-doc and overclaim search:
- Commands run:
- Terms/surfaces searched:
- Findings removed:
- Remaining intentional caveats:

Exact gates:
- Command:
  Result:
  Evidence summary:
- Command:
  Result:
  Evidence summary:

Caveats and deferred work:
- Caveat:
  Owner-visible rationale:
  Follow-up card or explicit non-goal:

No-overclaim statement:
- This closeout claims only the verified local/demo/review-gated behavior named above. It does not claim production deployment, live customer/provider side effects, or unresolved follow-up work.
```

## Mechanical guardrail

Run the workspace-quality guardrail before closeout:

```sh
python scripts/check_workspace_quality.py --repo-root .
```

The guardrail checks:

- changed Markdown for stale-process wording that should either be removed or explicitly justified in the handoff;
- tracked and untracked cache/build/database/archive artifacts that should not survive as workspace noise;
- the checked OpenAPI v0 artifact against the literal Axum `/v0/` route surface.

The guardrail is intentionally narrow. Passing it does not replace targeted Rust, web, docs, compose, or smoke gates required by the card; it only proves that common stale/noisy closeout failures were checked mechanically.
