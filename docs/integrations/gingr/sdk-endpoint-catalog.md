# Gingr SDK read-only endpoint catalog

Sources: Gingr Developer Guide local corpus, primarily `articles/25722122517517-gingr-api-functions-reference.md`, plus API-key/read-only overview articles. No live Gingr calls or real API keys were used.

## Boundary and transport rules

- Base URL pattern: `https://{your_app}.gingrapp.com`.
- Transport: HTTPS over TLS; responses are documented as JSON objects.
- Authentication: every `/api/v1` request uses a user-account-specific `key` parameter unless noted otherwise. The Gingr user must have `Can Access API` permission.
- Key redaction: never log, print, persist, or fixture real API keys. SDK errors, URL displays, tracing, request debug output, and tests must render `key` as `<redacted>`.
- Read-only boundary: Gingr docs state the public API is read-only; SDK v0 should still exclude documented side-effect endpoints and expose only typed request/query builders.
- Encoding: examples use `application/x-www-form-urlencoded`; SDK should encode GET query parameters and POST form bodies without treating POST as mutation by default.

## SDK module groups

- `reservations`: `reservations`, `reservation_widget_data`, `reservations_by_animal`, `reservations_by_owner`, `reservation_types`, `get_services_by_type`, `existing_reservation_estimate`, `recently_cancelled_reservations`, `back_of_house`
- `owners_animals`: `authorize_owner`, `new_modified_owners`, `forms_get_form`, `owner`, `owners`, `animals`, `custom_field_search`, `get_feeding_info`, `get_medication_info`
- `reference_data`: `get_locations`, `get_species`, `get_breeds`, `get_vets`, `get_temperaments`, `get_immunization_types`, `get_animal_immunizations`
- `commerce_retail`: `get_all_retail_items`, `list_transactions`, `list_invoices`, `transaction`, `get_subscription`, `get_subscriptions`
- `labor_ops`: `timeclock_report`
- `report_cards_files`: `report_card_files`

## Excluded from read-only SDK v0

| Endpoint | Method/path | Why excluded |
|---|---|---|
| `quick_checkin` | GET `/api/v1/quick_checkin` | Side effects: checks in pets, creates a reservation if one does not exist, and checks it in. |
| `receive_call` | POST `/api/v1/receive_call` | Side effects: triggers in-app alert and records an incoming phone call. |

## Implementation notes for SDK builders

- Represent `key` as a secret wrapper and require redacted display/debug implementations.
- Model date formats explicitly: `YYYY-MM-DD` for Gingr date params; ISO 8601 only where the reference names ISO 8601 (`reservations_by_animal` / `reservations_by_owner` nested params).
- Enforce documented range/pagination invariants in builders: reservations max 30-day range; `list_invoices` requires `per_page` and `page` together and uses result-offset-style page values; `get_subscriptions` paginates with `limit`/`offset`.
- Treat location scope as part of request semantics, not a cosmetic filter. Several endpoints are limited by current API-user location or explicit `location_id`.
- Quarantine raw provider responses before mapping to domain concepts; `custom_field_search`, owner/animal filters, transaction/payment details, and auth checks may contain high-sensitivity PII or credentials.
- Keep `quick_checkin` and `receive_call` out of the read-only SDK surface unless a separate human-reviewed side-effect design is approved.
