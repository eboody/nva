# Entity contract surface inventory

Purpose: evidence discovery for later entity-atlas contract crosswalks. This is not a product-doc rewrite. It inventories the concrete repo surfaces where NVA pet-resort entities appear and records whether each surface is an authoritative source, boundary adapter, workflow packet, persisted projection, runtime shell, test proof, or documentation/Rustdoc view.

Crosswalk back-links: after finding a surface here, use [source/provider flows](source-provider-flows.md) for provider entry and normalization, [workflow packets](workflow-packets.md) for app workflow use, [storage/persistence](storage-persistence.md) for persistence or non-persistence evidence, [runtime exposure](runtime-exposure.md) for API/worker/CLI/web/script exposure, and the [entity index proof paths](../../design/entity-index.md#contract-crosswalk-proof-paths) to return to entity pages.

Scope inspected:

- Source contracts and provider payloads under `domain/src`, `integrations/gingr/src`, `app/src`, `storage/src`, and `apps/*`.
- Gingr DTOs, endpoint request builders, raw response records, webhook envelopes, and mapping modules.
- App workflow packets, safe/blocked action enums, tool ports, API state/routes, worker runtime config, and CLI inspection shell.
- Storage records/projections, migration tables, tests/fixtures, README/docs, and source Rustdocs.
- Existing entity-atlas drafts under `docs/design/entity-atlas-*.md`, especially the seed inventory and runtime/storage page.

Reading rule for this inventory:

- "Authoritative source" means the surface owns the contract for that layer. It does not always mean "business truth". Example: Gingr endpoint types are authoritative for provider request shape, while `domain::*` is authoritative for normalized business meaning.
- "Derived view" means the surface should cite or convert from another contract rather than invent entity meaning.
- Evidence links point at repo files/lines from this workspace. Rendered Rustdoc HTML was not regenerated during this pass, so Rustdoc evidence is source-doc evidence.

## Crosswalk notes for later entity-page writers

1. Provider facts flow from Gingr endpoint/response/DTO/webhook surfaces into mapping candidates and source refs. They should not be described as NVA truth until a mapper/domain contract says what was promoted.
2. `domain::*` owns semantic entity meaning; `app::*` owns workflow packet/review/draft/outcome behavior; `storage::*` owns durable projection/code shape; `apps/*` owns runtime shell exposure.
3. The same entity often appears in multiple forms: provider owner/animal/reservation ids, domain `Customer`/`Pet`/`Reservation`, app workflow packet fields, SQL rows, storage projections, API JSON DTOs, and docs/glossary pages. Later crosswalks should name the form, not flatten them into one vague term.
4. Review gates, blocked actions, audit events, source refs, and outcome records are contract surfaces, not implementation details. They are how the repo preserves safety while reducing manual manager/front-desk work.
5. Tests and fixtures should be linked as proof near each final entity page, but they should not become the entity definition unless the test is explicitly naming a fixture-only scenario.
