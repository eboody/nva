# Gingr/source-provider normalization crosswalk

This page helps front desk leads, managers, data-quality reviewers, and engineering maintainers avoid repeated Gingr/source reconciliation by showing which provider DTOs enter the repo, which mapper or source-contract normalizes them, what remains only provider evidence, and which tests prove the narrow mapping. It is a crosswalk, not a new source of product authority: live customer sends, Gingr/provider writes, schedule/check-in/out changes, payment movement, and safety approvals remain blocked unless an app workflow and human review gate explicitly allow them.

Use this page with the broader [Gingr provider boundary atlas](../../integrations/gingr/provider-boundary-atlas.md), [source/provenance/data-quality atlas](../../design/source-provenance-data-quality-atlas.md), and [entity relationship map](../../design/entity-atlas-relationships.md). Source files and generated Rustdoc remain authoritative for behavior.

Crosswalk back-links: use [surface inventory](surface-inventory.md) to find source/Rustdoc/test locations before this normalization step, [workflow packets](workflow-packets.md) to see where normalized source evidence is used, [storage/persistence](storage-persistence.md#source-provenance-data-quality-and-provider-boundary-entities) for durable source refs and provider-boundary projections, [runtime exposure](runtime-exposure.md#exposure-table) for API/CLI/script exposure, and the [entity index proof paths](../../design/entity-index.md#contract-crosswalk-proof-paths) to return to business-language entity pages.

## Boundary flow at a glance

```text
Gingr API / webhook / fixture / import evidence
  -> integrations/gingr endpoint, response, webhook, or DTO type
  -> mapper, source::gingr promotion, or explicit ProviderSurface gap
  -> domain source/provenance value or reviewable domain candidate
  -> app workflow packet, storage projection, data-quality issue, or BI/stay projection
  -> human review/outcome record before any live business action
```

Important authority rule: Gingr is authoritative only for the provider record it returned or event it signed. NVA domain/app/storage contracts decide whether that record becomes a customer contact candidate, pet-name candidate, retail product candidate, reservation snapshot, data-quality issue, workflow queue item, or stored outcome evidence.

- Webhook fixtures/tests: `integrations/gingr/tests/webhook_contracts.rs` and `docs/integrations/gingr/fixtures/webhooks/README.md` cover signed/unsigned webhook behavior and fixture scope.

## Caveats and missing evidence

1. There is no direct `integrations/gingr` mapper from `ReservationRecord` to `domain::source::gingr::reservation::Snapshot`; reservation normalization currently lives in `domain::source::gingr` contracts and tests, while the Gingr response DTO only preserves provider fields.
2. `response::ReferenceRecord` is modeled as a provider DTO, but this pass found no mapper that promotes locations, vets, immunization types, or other reference rows into domain entities.
3. Grooming and training provider surfaces intentionally remain `ProviderSurface::NoDocumentedServiceDto` gaps. Domain grooming/training contracts exist, but this codebase does not yet prove a Gingr service DTO-to-domain mapping for them.
4. Retail quantity-on-hand is retained on the Gingr DTO but not carried into `ProductCandidate`; inventory reconciliation remains downstream/future evidence.
5. Webhook verification is well modeled, but the inspected source does not prove a webhook-payload-to-domain-entity mapper. Treat verified webhooks as source event evidence until a mapper names the resulting domain fact.
6. App workflow tests use `source::Provenance` with Gingr endpoints/record ids rather than raw `gingr::response::*` DTOs; that proves source-consumption boundaries, not end-to-end live extraction.
7. `StoredSourceRecordRef` keeps durable source pointers as strings for storage projection; it does not by itself prove raw payload validity or live provider state.
8. Many docs and source files are currently uncommitted on this working tree. This page links to their present paths and source contracts; future workers should re-run link/docs checks after the surrounding atlas/doc changes settle.

## Reviewer checklist for future mapping claims

Before claiming a source-provider field is normalized, verify all of the following:

- The provider DTO or source-provider type is named and linked.
- The mapper/promotion function is named, not implied by endpoint presence.
- The normalized entity/record is a domain/app/storage contract, not a provider id renamed as truth.
- Source authority is scoped: "Gingr says" vs. "NVA derives" vs. "human approves".
- Lossy/derived behavior is explicit: trimming, category collapse, unknown-field retention, status assumptions, and any missing-field error instead of silent defaults.
- A test, doctest, fixture, or Rustdoc page proves the exact mapping.
- Missing evidence remains a caveat or `ProviderSurface`/data-quality gap rather than prose authority.
