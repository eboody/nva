# Runtime exposure crosswalk

Purpose: map the runtime surfaces that expose, mutate, import, export, or schedule entity data today. Use this as the non-coder checklist before describing any API, worker, CLI, app shell, smoke script, bridge script, webhook, or integration surface as operational automation.

Crosswalk back-links: use [surface inventory](surface-inventory.md) for source/Rustdoc/test locations, [source/provider flows](source-provider-flows.md) for provider entry and normalization, [workflow packets](workflow-packets.md#workflow-by-workflow-entity-map) for app workflow use, [storage/persistence](storage-persistence.md) for persisted or intentionally mock-only state, and the [README entity navigation map](../../../README.md#entity-navigation-map) or [operator workflow index](../../workflows/operator/README.md) to return to non-coder entry points.

Scope notes:

- "Entity data" means pet-resort entities and workflow evidence: customer/owner, pet/animal, reservation, inquiry, document/vaccine, manager action, data-quality issue, incident, task, staff/user, source refs, audit events, outcome records, and integration/provider ids.
- Most current surfaces are deterministic local/demo/contract surfaces. They expose packets, drafts, review queues, and outcome evidence; they do not grant live customer, provider/PMS, schedule, payment, or safety authority.
- The required human-review boundary is part of the runtime contract. If a surface lacks a reviewer or fails closed, keep that visible in public docs and operator docs.
- App shell naming mismatch to call out: Rust binaries/packages are `pet-resort-api`, `pet-resort-worker`, and CLI command `pet-resort`; the web shell is the npm workspace `@pet-resort/staff-web` under `apps/staff-web`; Docker Compose services are `pet-resort-api` and `pet-resort-worker`; API env uses `PET_RESORT_API_ADDR` while bridge scripts use `PET_RESORT_API_URL`.

## Gaps and mismatches for non-coders

1. No durable queue/scheduler job is implemented yet in `apps/worker`; it is a safe process shell with fake/disabled agent runtime and stubbed side effects.
2. No live import/export CLI exists today. The CLI exports JSON inventory only (`agents`, `tools`), and Gingr import/export-like surfaces are library request/webhook parsers with fixture/mock tests.
3. The API README may lag the router when new route groups are added; treat `apps/api/src/http.rs` and `apps/api/tests/*` as the route authority.
4. The staff web app is an npm/Next local MVP shell, not an API-backed source of truth. It uses local static/demo data to display review gates and blocked actions.
5. `PET_RESORT_API_ADDR` is for the server bind address; `PET_RESORT_API_URL` is for bridge/client scripts. Confusing the two is a common setup mistake.
6. `pet-resort` is the CLI command name, not the API or worker binary name. Cargo package names are `pet-resort-api`, `pet-resort-worker`, and `cli`.

## Safe wording to reuse

- "This surface exposes reviewable packets/drafts/outcome evidence; it does not execute live side effects."
- "Source facts remain source refs until an app/domain/storage contract promotes them with review evidence."
- "Accepted draft means app validation passed, not that a customer was contacted or a provider record changed."
- "Outcome captured means reviewed labor evidence was recorded locally/storage-shaped; it is not proof of a live PMS/payment/schedule mutation."
