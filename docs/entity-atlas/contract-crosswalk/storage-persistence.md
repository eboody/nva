## Executive storage map

Current persistence has two distinct layers that docs must keep separate:

- [`migrations/0001_mvp_foundation.sql`](../../../migrations/0001_mvp_foundation.sql) defines the MVP SQL schema for launch-critical owners, pets, reservations, documents, vaccine evidence, review packets, approvals, workflow events/results, outbox/object metadata, payment/deposit projections, and append-only audit surfaces.
- The `storage` crate still owns Rust storage records, code enums, JSON codecs, and executable tests for the current storage projection contracts; those Rust records are narrower than the SQL migration and are not a live database adapter by themselves.
- `storage` owns persistence projections and JSON codec boundaries. It is not domain truth and does not authorize live provider/customer/payment/schedule actions.
- `apps/api` uses deterministic in-memory state (`BTreeMap`/`Vec`) for demos and contract tests. That state is mock/runtime shell persistence only, not a production repository.
- `integrations/gingr` owns provider DTOs, endpoint request builders, raw envelopes, and verified webhooks. Those surfaces preserve provider evidence; they are not local storage tables.
- `app` owns workflow packets, validation, draft/outcome semantics, and tool/repository ports. Most app packets are computed/read-only packet state, not persisted entities.

## Per-entity persistence rows

Columns:

- `Persistence table/record/projection`: local storage/API/provider projection currently visible.
- `Authority`: source of truth vs cache/derived/mock-only.
- `Repository/store paths`: code paths that store, project, or port the entity.
- `Read/write workflow`: how it is read or written today.
- `Exposure paths`: API, CLI, provider, docs, or runtime surface.
- `Evidence`: tests/docs/source evidence.
- `Retention/audit caveats`: important caveats for docs writers.

## Absent or intentionally mock-only storage surfaces

- A SQL MVP migration exists, including outbox and audit tables, but this crosswalk did not find API/worker repository adapters proving those tables are used by the runtime shells today.
- No ORM models, queue workers/tables wired to runtime, object-storage adapter, user directory, auth store, or provider-write ledger are present in this repo slice.
- Core domain entities such as full `Location`, `Customer`, `Pet`, `Reservation`, `CareProfile`, `Incident`, `Message`, and staff/user directories are semantic domain contracts, not locally persisted tables.
- API vaccine/document/inquiry/approval/outcome state is intentionally in-memory for deterministic contract tests and demos.
- Provider entities are held as DTOs, endpoint request descriptors, mapping candidates, webhooks, and raw/envelope records under `integrations/gingr`; they remain provider evidence until promoted through app/domain/storage boundaries.
